# ADR-0004: 州系统作为元数据规范

> **决策状态**：采纳
> **提出日期**：2026-04-12
> **提出者**：设计团队
> **关联文档**：`design/gdd/state-system.md`, `design/gdd/map-design.md`, `design/gdd/cards-design.md`

## 1. 决策背景

项目已设计出十三州的地理、战役、专属卡和兵种联动机制，但这些内容分散在 `map-design.md` 和 `cards-design.md` 中，缺乏统一的配置规范。这可能导致：

- 设计数据不一致（如州专属卡数量在不同文档中冲突）
- 工程实现时难以准确定位州相关配置
- 新成员难以理解州系统的整体架构

## 2. 考虑的替代方案

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| **1. 创建独立州系统** | 将州作为独立游戏系统，包含专属机制和状态 | ❌ 违反设计原则：州是配置属性而非机制，会过度设计，增加复杂度 |
| **2. 保留分散设计** | 保持当前状态，州相关内容分别存在于地图和卡牌GDD中 | ❌ 难以维护一致性，容易出现数据冲突 |
| **3. 创建州元数据规范（采纳）** | 创建独立ADR文件，统一定义州的配置属性和数据关系 | ✅ 保持架构清晰，不引入新机制，仅提供配置标准，易于维护 |

## 3. 决策内容

我们决定：

1. **创建 `design/gdd/state-system.md` 作为州系统的元数据规范**
2. **州不是独立的游戏系统，而是地图节点系统和卡牌解锁系统的配置属性**
3. **所有州相关信息（地形、战役、专属卡、兵种联动）必须在此文件中统一定义**
4. **`map-design.md` 和 `cards-design.md` 仅引用此文件，不再重复定义州信息**
5. **"涉及武将"字段仅列出在该州有战役记录的武将，不区分主次**
6. **每位武将的生涯战役跨越多个州，一个州可出现在多个武将的战役中**
7. **州专属卡数量（共143张）为设计目标，解锁条件为通关包含该州小地图的战役**

## 4. 决策依据

- **简化架构**：避免在核心系统中嵌入配置逻辑，保持游戏系统（地图、卡牌）与数据（州）的分离
- **增强一致性**：确保所有设计师和程序员使用同一个数据源，避免信息孤岛
- **符合设计原则**：遵循GDD标准，每个系统只负责其核心机制，配置数据由独立文件管理
- **便于扩展**：新增州或修改州配置时，只需修改单一文件
- **准确表达设计意图**：明确州是地理背景，非战役归属；武将的生涯战役跨越多个州

## 5. 对现有设计的影响

- ✅ `map-design.md`：删除所有州清单和专属卡数量说明，仅保留地图结构和节点逻辑
- ✅ `cards-design.md`：删除州专属卡分类说明，仅保留卡牌效果，州归属指向 `state-system.md`
- ✅ `hero_campaign_maps.csv`：保持不变，作为 `state-system.md` 的数据实现
- ✅ 所有团队成员需在修改州相关信息时，优先更新 `state-system.md`
- ✅ "涉及武将"字段基于 `hero_campaign_maps.csv` 的数据统计，动态更新

## 7. 结论

通过将州作为元数据规范而非独立系统，我们以最小成本实现了设计一致性，同时保持了架构的清晰性和可维护性。这一决策体现了"配置与机制分离"的最佳实践。
