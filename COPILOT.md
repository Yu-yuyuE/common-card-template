# COPILOT.md（项目级指令）

此文件是 **Copilot Code Pro** 的项目级指令入口，功能上等价于 Claude Code 的 `CLAUDE.md`。  
它用于统一约束本仓库中的 AI 协作流程、角色分工、文档标准与设计决策方式；读者无需了解 Claude Code 也可直接使用本文件。

---

## 项目概述

这是一个卡牌游戏设计与开发项目（当前仓库：`d:/develop/_godot/common-card-template`），采用“工作室化多角色协作”方法推进游戏设计、系统拆解与实现。

核心目标：

- 用明确分工提升设计与实现质量
- 通过标准化文档沉淀决策与知识
- 通过可追溯流程降低返工与沟通成本

## 角色体系与职责边界（工作室分层）

本项目采用三层角色组织：

1. **Tier 1（总监层）**：创意、技术、制作方向把控
2. **Tier 2（部门负责人）**：设计、程序、美术、音频、叙事、QA、发布、本地化等领域统筹
3. **Tier 3（专家执行层）**：在具体子领域产出可执行方案、文档、实现与验证结果

角色全表与适用场景请参考：`.copilotcode/docs/agent-roster.md`。

### 协作原则

- 工作应沿层级向下分解（方向 -> 方案 -> 执行）
- 冲突向上升级处理（专家 -> 负责人 -> 总监）
- 跨部门任务由制作/编排角色统筹
- 任何关键结论必须文档化，避免口头决策遗失

## Copilot 等价机制映射（替代 Claude 专属机制）

为保证语义一致但工具链可执行，采用以下等价规则：

1. **技能调用（替代 `/skill` slash commands）**  
   - Copilot Chat **不会像 Claude Code 一样动态读取自定义 slash command**，因此直接输入 `/brainstorm`、`/team-combat` 这类命令，通常**不会自动触发项目技能**。  
   - 本项目的技能入口统一迁移到 `.copilotcode/skills/`，每个技能位于对应目录下的 `SKILL.md`。  
   - Copilot Code Pro 会在运行时执行强制技能匹配（mandatory skill check），因此要用**自然语言显式点名技能**，而不是依赖 `/`。  
   - 推荐触发方式示例：  
     - “使用 brainstorm 技能，帮我做一个 roguelike 卡牌游戏概念发散”  
     - “请使用 team-combat 技能，设计并推进战斗系统协作”  
     - “使用 design-review 技能，审查 `design/gdd/card-battle-system.md`”  
   - 如果你在 Copilot 插件或外部封装层里自行配置了 slash command，也应让其**映射到自然语言技能触发语句**，而不是假设 Copilot 会自动扫描本仓库目录。

2. **向用户追问（替代 `AskUserQuestion`）**  
   - 统一使用 `ask_followup_question` 工具。  
   - 问题需聚焦缺失信息，提供 2–4 个可直接选择的建议答案。

3. **子代理委派（替代 `Task` 子代理）**  
   - 使用 `new_task` 进行编排器级委派。  
   - 在跨领域复杂任务中，将子任务分发到合适模式（如 architect / code / debug）并维护清晰的任务边界与交付物。

4. **角色定义目录映射（替代 `.claude/agents/`）**
   - 角色知识已迁移至 `.copilotcode/agents/`。
   - 在 Copilot Code Pro 中，实际的协作编排优先通过技能说明、模式切换与任务委派实现。

5. **规则目录映射（替代 `.claude/rules/`）**
   - 规则知识已迁移至 `.copilotcode/rules/`。
   - 执行前先匹配规则，避免越权修改与不合规输出。

6. **Hook 机制映射（替代 `.claude/hooks/`）**
   - 采用“编排器事件 + CI hooks”组合实现自动检查、门禁与审计。
   - 例如：在任务阶段切换、PR 校验、发布流程中触发一致性检查。

7. **会话状态持久化（保持不变）**  
   - `production/session-state/active.md` 作为进行中会话状态的持久化入口，机制保持不变。  
   - 需要在关键节点记录：当前目标、已完成项、阻塞项、下一步动作。

### 常用技能触发示例

- 概念发散：`使用 brainstorm 技能`
- 系统拆解：`使用 map-systems 技能`
- 单系统 GDD 编写：`使用 design-system 技能`
- 设计审查：`使用 design-review 技能`
- 项目阶段诊断：`使用 project-stage-detect 技能`
- 战斗团队编排：`使用 team-combat 技能`
- UI 团队编排：`使用 team-ui 技能`
- 发布前检查：`使用 release-checklist 技能`

### 技能目录说明

当前项目技能已按 Copilot Code Pro 结构迁移到 `.copilotcode/skills/`，目录形式如下：

- `.copilotcode/skills/{skill-name}/SKILL.md`

每个 `SKILL.md` 至少包含：

- `name`：技能名
- `description`：适用场景描述，供 mandatory skill check 匹配
- `argument-hint`：建议参数格式
- 详细执行步骤、输入约束、输出格式与后续建议

## 标准工作流（用户驱动协作）

**原则：用户驱动协作，而非自主闭环执行。**

所有任务遵循：

**问题澄清 -> 方案选项 -> 用户决策 -> 草案产出 -> 用户确认**

执行约束：

- 在写入或修改文件前，应先明确将写入哪个路径与写入意图
- 在请求最终确认前，应提供草案摘要或变更说明
- 多文件改动应以“完整变更集”维度进行确认
- 未获明确指令，不进行提交、发布或不可逆操作

协作原则详见：`docs/COLLABORATIVE-DESIGN-PRINCIPLE.md`

## 新项目/新阶段启动建议（Copilot 语境）

当项目尚未明确引擎、概念或阶段时，建议按以下次序执行：

1. **阶段识别与缺口盘点**：先做项目现状盘点（文档、原型、代码、里程碑）
2. **概念生成或校准**：必要时进行结构化头脑风暴，形成游戏概念文档
3. **引擎与版本落定**：确认引擎版本、技术约束与性能预算
4. **系统拆解**：形成系统清单、依赖关系与优先级
5. **逐系统设计**：按依赖顺序编写/完善 GDD
6. **原型与验证**：先验证核心循环，再扩展产能
7. **测试与评审**：引入可执行测试计划和阶段门禁
8. **迭代计划**：进入里程碑与 Sprint 节奏

> 实操时，优先通过对应技能文档（`.copilotcode/skills/*/SKILL.md`）加载具体步骤模板。

## 文档标准与输出要求

### 1) 文档先行，决策可追溯

- 关键设计与技术决策必须落文档
- 文档应注明上下文、约束、方案比较与结论
- 避免仅在会话中“口头定案”

### 2) 结构化输出

- 长文档遵循固定章节（背景、目标、约束、方案、风险、结论）
- 系统设计需包含数据结构、状态流、边界条件、失败处理
- 数值与平衡设计需提供假设与校验方法

### 3) 一致性维护

- 新文档需与现有 GDD、系统索引、术语保持一致
- 修改既有机制时，应同步更新关联文档并注明影响范围

## 协作协议（跨角色/跨模式）

1. **明确主责角色**：每个任务先确定最终负责方
2. **按专长委派**：跨领域问题由编排器拆分并委派子任务
3. **统一回收**：由主责角色汇总结论，形成单一对外口径
4. **冲突升级**：创意冲突交创意负责人，技术冲突交技术负责人，排期冲突交制作负责人
5. **保留审计线索**：记录关键决策时间点、依据与影响

---

## 会话上下文与状态管理

- 上下文预算有限，需优先保留：当前目标、硬约束、最近决策、阻塞项
- 大体量材料应进行摘要化与索引化，而非无界粘贴
- 活跃会话状态统一维护在：`production/session-state/active.md`（机制不变）

建议最小记录字段：

- 当前阶段/目标
- 今日完成
- 当前阻塞
- 待决策问题
- 下一步（可执行动作）

---

## 执行边界与质量门禁

- 不越过职责边界修改无关领域内容
- 高风险变更先评审再落地
- 对外可见内容（发布说明、公告、Patch Notes）需经过一致性校对
- 阶段推进前执行门禁检查（设计完整性、实现可行性、测试覆盖、发布准备度）

## 快速使用说明（Copilot）

1. 先阅读本文件，确认当前阶段与目标
2. 按任务性质选择合适模式（或用 `new_task` 委派）
3. 若命中专项流程，先读取对应 `.copilotcode/skills/*/SKILL.md`
4. 按“问题 -> 选项 -> 决策 -> 草案 -> 确认”推进
5. 将会话关键状态同步到 `production/session-state/active.md`

## 结语

本文件定义了本项目在 Copilot Code Pro 下的统一协作契约：

- 保留原有工作室化分工与文档治理思想
