# 本地化系统

> **状态**: 设计中
> **作者**: [用户 + 代理]
> **最后更新**: 2026-04-09
> **实现支柱**: 历史主题, Roguelike清晰度

## 概述

本地化系统是一个基础支撑性服务，它使游戏能够支持多种语言（中文、英文、日文），同时在所有语言版本中保持一致的策略体验。

作为一个数据/基础设施层，该系统管理所有游戏文本资源的翻译、存储和运行时切换。它确保每个UI元素、卡牌描述、系统消息和叙事文本都能以玩家偏好的语言显示，而不影响游戏机制或平衡性。

对玩家而言，该系统提供"语言即工具"的体验幻想 - 能够在保持策略决策专注的同时无缝切换语言。无论玩家选择中文的历史真实性、英文的广泛清晰度，还是日文的文化视角，"运筹帷幄之中，决胜千里之外"的核心体验都将保持不变。该系统消除了语言作为入门障碍，同时尊重每种文化表达战略概念的独特方式。

该系统集成了Godot 4.6内置的本地化框架，使用基于CSV的翻译文件、按语言组织、自动字符串提取和运行时语言切换功能。它遵循项目已建立的UI数据绑定模式(ADR-0016)，确保文本更新能够正确传播到整个界面层次结构中。

## 玩家幻想

本地化系统服务于一种间接的玩家幻想 - 它作为透明的基础设施存在，消除语言作为战略专注的障碍。玩家主要通过系统的"不存在"来体验它；当它完美工作时，语言在关键决策时刻永远不会成为考虑因素。

**核心幻想："语言即隐形桥梁"**

玩家应该感受到无缝文化穿越的力量 - 能够随意切换语言而不打断战略流状态的能力。该系统赋予了成为真正国际化战略家的幻想，一个能够通过多语言视角参与同一深度战术内容而毫无摩擦的战略家。

**关键玩家时刻：**

1. **入场掌控** - 首次启动时，玩家发现偏好的语言已经可用，立即产生舒适感和归属感
2. **激战切换** - 在激烈战斗过程中，玩家可以即时切换语言以更好理解复杂卡牌描述，而不打断他们的节奏
3. **文化视角转换** - 玩家发现同样的战略概念在不同语言中有不同表达，丰富他们对三国宇宙的理解
4. **社区包容性** - 玩家可以与使用不同语言偏好的朋友讨论策略，知道他们体验的是同一核心内容

**情感目标：**

- **信心** - 信任所有游戏内容在所有支持语言中都能同等获取
- **包容性** - 感受到他们的语言选择受到尊重和良好实现
- **专注** - 确信语言永远不会成为游戏障碍的安全感
- **文化欣赏** - 认识到每种语言都为战略叙事带来独特的表达优势

该系统的成功不是以玩家对它的参与度来衡量的，而是以玩家对它的无感知来衡量的。当完美实现时，玩家将只在主动注意时才注意到本地化 - 并且会对无缝体验感到愉悦。

## 详细设计

### 核心规则

1. **文本-逻辑分离**：所有显示文本必须通过键值引用与代码解耦。源代码中不允许有硬编码字符串。

2. **运行时语言切换**：玩家可以在游戏过程中随时切换语言。系统必须立即刷新所有UI元素和文本节点，而无需重新加载场景或重启游戏。

3. **语言优先级回退**：如果当前语言缺少特定键的翻译，系统将自动回退到预定义的"主要语言"（中文）。这确保文本不会显示为空白或键名。

4. **上下文感知文本替换**：关键文本元素如角色名、单位类型和地形名必须支持参数化替换（例如，"{general_name} 激活了 {skill_name}"），以确保语法结构能适应不同语言。

5. **字体和布局适配**：不同语言使用不同字体（中文：思源黑体；英文：Roboto；日文：Noto Sans CJK JP）。系统根据语言自动选择字体资源，并支持不同的字符宽度和行高。

### 管理状态和转换

本地化系统作为状态机运行，具有以下状态：

| 状态 | 描述 | 触发条件 |
|------|------|----------|
| `INITIALIZING` | 系统启动，加载语言包 | 游戏启动，语言设置更改 |
| `LOADED` | 语言包已加载，文本服务可用 | 所有语言文件成功解析 |
| `FALLBACK_ACTIVE` | 部分键缺失，回退机制激活 | 检测到缺失键且主要语言可用 |
| `ERROR` | 语言包损坏或缺失 | 无法解析JSON或缺少主要语言文件 |
| `SWITCHING` | 语言切换进行中，UI刷新中 | 玩家调用 `set_language()` |

**状态转换**：
- INITIALIZING → LOADED (加载成功)
- INITIALIZING → ERROR (加载失败)
- LOADED → SWITCHING → LOADED (语言切换成功)
- LOADED → SWITCHING → FALLBACK_ACTIVE (切换后部分键缺失)
- FALLBACK_ACTIVE → LOADED (缺失键已填充)
- ERROR → INITIALIZING (用户重试或重新选择语言)

### 与其他系统的交互

| 系统 | 交互方式 | 数据流向 | 注意事项 |
|------|----------|----------|----------|
| **卡牌系统** | 通过 `Localization.get("card.swordmaster.desc")` 获取卡牌描述和名称 | 本地化 → 卡牌系统 | 卡牌模板必须存储语言键而非文本，以避免运行时拼接 |
| **地图系统** | 通过键调用节点事件文本（例如，"张飞伏击"） | 本地化 → 地图系统 | 事件配置文件应引用语言键，例如，`"event_text": "event.ambush_zhangfei"` |
| **UI系统** | 所有按钮、标签和工具提示都绑定到语言键 | 本地化 → UI系统 | UI控件必须监听 `language_changed` 信号并刷新 |
| **存档系统** | 保存当前语言设置 (`current_language`) | 存档 ← 本地化 | 语言设置是存档数据的一部分，加载时自动恢复 |
| **音频系统** | 语音文件名必须与语言键匹配（例如，`voice_card_swordmaster_en.mp3`） | 本地化 → 音频系统 | 语音文件命名约定：`voice_[type]_[key]_[lang].mp3` |

## Formulas

### F1. 文本检索公式

`LocalizedText = GetTextByKey(Key, Language)`

**变量：**
| 变量 | 符号 | 类型 | 范围 | 描述 |
|------|------|------|------|------|
| 键 | key | 字符串 | 任意有效字符串键 | 文本字符串的唯一标识符 |
| 语言 | lang | 字符串 | "zh", "en", "ja" | 目标语言代码 |
| 本地化文本 | result | 字符串 | 任意字符串 | 目标语言中的本地化文本 |

**输出范围：** 返回给定键的本地化字符串，如果键缺失则返回回退字符串。
**示例：** `GetTextByKey("ui.button.start", "zh")` 返回 "开始游戏"

### F2. 语言切换性能

`SwitchTime = BaseLoadTime + (TextCount × ProcessingOverhead)`

**变量：**
| 变量 | 符号 | 类型 | 范围 | 描述 |
|------|------|------|------|------|
| 基础加载时间 | T_base | 浮点数 | 10-50ms | 初始化语言切换的基础时间 |
| 文本数量 | N_text | 整数 | 1-10000 | 要更新的文本元素数量 |
| 处理开销 | T_overhead | 浮点数 | 0.1-1.0ms | 每个文本元素的处理时间 |
| 切换时间 | T_switch | 浮点数 | 20-200ms | 切换语言的总时间 |

**输出范围：** 正常情况下 20ms 到 200ms；重负载下可达 500ms。
**示例：** 切换语言包含 1000 个文本元素：30ms + (1000 × 0.5ms) = 530ms

### F3. 语言包内存使用

`MemoryUsage = FixedOverhead + (LanguageCount × AveragePackSize)`

**变量：**
| 变量 | 符号 | 类型 | 范围 | 描述 |
|------|------|------|------|------|
| 固定开销 | M_fixed | 整数 | 1-10MB | 本地化系统的基础内存 |
| 语言数量 | N_lang | 整数 | 1-10 | 支持的语言数量 |
| 平均包大小 | M_pack | 整数 | 1-5MB | 一个语言包的平均大小 |
| 内存使用 | M_total | 整数 | 5-50MB | 本地化系统使用的总内存 |

**输出范围：** 5MB 到 50MB，取决于语言数量和内容大小。
**示例：** 使用 3 种语言和 3MB 平均包大小：5MB + (3 × 3MB) = 14MB

### F4. 字体加载时间

`FontLoadTime = BaseFontTime + (CharacterSubsetSize × LoadFactor)`

**变量：**
| 变量 | 符号 | 类型 | 范围 | 描述 |
|------|------|------|------|------|
| 基础字体时间 | T_font_base | 浮点数 | 50-200ms | 加载基础字体信息的时间 |
| 字符子集大小 | N_chars | 整数 | 1000-10000 | 子集中的字符数量 |
| 加载因子 | F_load | 浮点数 | 0.01-0.1ms | 每个字符的加载时间 |
| 字体加载时间 | T_font | 浮点数 | 100-1000ms | 加载字体资源的总时间 |

**输出范围：** 100ms 到 1000ms，取决于字体复杂度和字符集大小。
**示例：** 加载包含 5000 个 CJK 字符的字体：100ms + (5000 × 0.05ms) = 350ms

## 边缘情况

1. **如果当前语言缺少翻译键**：系统将自动回退到主要语言（中文）。如果主要语言也缺少该键，系统将显示"[MISSING: key_name]"来指示缺少的翻译。

2. **如果在战斗中发生语言切换**：系统将所有UI刷新请求排队，并在下一帧处理它们，以避免中断战斗动画或逻辑更新。所有文本元素包括技能冷却时间、伤害数值和状态效果都会原子性地刷新。

3. **如果特定语言的字体文件加载失败**：系统将回退到默认的Unicode兼容字体（如Noto Sans），该字体可以显示所有支持的语言，尽管视觉质量会有所降低。系统会记录错误供开发人员诊断。

4. **如果动态文本生成遇到不支持的参数**：对于像"{general_name} 使用 {skill_name}"这样的参数化文本，如果参数为null或无效，系统将用占位符如"[未知武将]"替换它，而不是崩溃或显示空白文本。

5. **如果文本扩展超出UI容器边界**：当翻译文本长于分配的UI空间时（常见于英文翻译比简洁的中文文本更长），系统将自动减小字体大小最多20%，或启用带省略号的文本剪裁以保持布局完整性。

6. **如果语音文件和文本不同步**：系统维护一个在游戏启动时验证的语音-文本映射表。不匹配的对将被记录为警告，系统将回退到仅文本显示该特定内容。

7. **如果特殊字符导致渲染问题**：系统包含一个Unicode标准化层，将有问题的字符（如全角空格或特殊标点符号）转换为在所有平台和字体上都能一致渲染的标准等效字符。

8. **如果语言包损坏或格式错误**：系统在加载时对语言包文件执行完整性检查。如果检测到损坏，系统将回退到主要语言，并向用户显示主要语言的错误消息，同时为开发人员记录详细的技术信息。

## 依赖关系

### 直接依赖

1. **UI系统** - 本地化系统直接与UI系统集成，为所有界面元素提供翻译文本。它遵循已建立的UI数据绑定模式(ADR-0016)，确保文本更新在整个界面层次结构中正确传播。所有UI元素必须绑定到语言键而不是静态文本。

2. **存档系统** - 本地化系统依赖存档系统来在游戏会话之间持久化玩家的语言偏好。当前语言设置作为用户偏好设置的一部分存储在存档数据中，并在加载存档时自动恢复。

3. **音频系统** - 对于语音功能，本地化系统与音频系统协作，确保语音文件与所选语言匹配。语音文件名遵循包含语言代码的特定命名约定。

4. **资源管理系统** - 本地化系统依赖资源管理系统来高效地加载和卸载语言包资源，确保最小的内存占用和对翻译字符串的快速访问。

### 依赖系统

1. **卡牌系统** - 所有卡牌名称、描述和风味文本都通过语言键由本地化系统提供。卡牌模板存储语言键而不是静态文本，以实现在运行时切换语言。

2. **地图系统** - 地图系统中的所有节点事件文本、地名和叙事描述都通过此系统进行本地化。事件配置文件引用语言键以确保跨语言的一致性。

3. **事件系统** - 所有事件描述、选择文本和结果消息都通过此系统进行本地化，以在所有语言中提供一致的体验。

4. **UI系统** - 所有用户界面元素包括按钮、标签、工具提示和菜单都依赖本地化系统提供其文本内容。UI系统必须监听语言更改事件并刷新所有文本元素。

### 系统集成要求

1. **ADR-0016 (UI数据绑定)** - 本地化系统必须遵循已建立的UI数据绑定模式，以确保与UI系统的无缝集成和文本更新的正确传播。

2. **ADR-0005 (存档序列化)** - 本地化系统必须与存档系统集成，以持久化语言偏好设置并确保在加载存档时正确恢复它们。

3. **Godot 4.6本地化框架** - 系统必须利用Godot 4.6的内置本地化功能，包括基于CSV的翻译文件、自动字符串提取和运行时语言切换。

## 调节参数

| 参数 | 安全范围 | 影响 |
|------|----------|------|
| `text_cache_size` | 1000–5000 条目 | 加载性能 | 内存使用 vs 文本加载延迟 |
| `fallback_language_order` | ["zh-CN", "en", "ja-JP"] 数组 | 缺失翻译恢复 | 键缺失时哪些语言作为回退 |
| `max_text_length_per_card` | 50–300 字符 | UI布局稳定性 | 截断前的最大可见字符数 |
| `auto_line_break_threshold` | 20–80 字符 | 可读性 vs 视觉密度 | 文本何时自动换到下一行 |
| `text_load_timeout_ms` | 100–1000 毫秒 | 响应性 vs 完整性 | 在回退前等待语言包加载的最长时间 |
| `dynamic_text_expansion_factor` | 1.5–2.5 | UI容器大小调整 | 语言间文本长度差异的乘数 |
| `use_localized_font_override` | {true, false} | 每种语言的视觉质量 | 是否为每种语言使用优化的字体 |
| `enable_text_replacement_rules` | {true, false} | 文化真实性 | 启用上下文感知的名称/标题替换 |
| `translation_update_check_interval` | 3600–86400 秒 | 内容新鲜度 | 检查更新翻译文件的频率（如果使用远程更新） |
| `font_size_scale_factor` | 0.8–1.2 | 可读性 | 每种语言的字体大小乘数 |

**调节原则：**
- **UI空间预算**：所有容器必须基于 `dynamic_text_expansion_factor * 1.2` 预留空间，以应对最坏情况的文本扩展。
- **内存 vs 性能**：`text_cache_size` 直接影响内存占用；根据总唯一键数设置（在游戏中使用调试覆盖层测量）。
- **回退链**：始终包含中文(`zh-CN`)作为最终回退，因为它是主要语言且最有可能有完整的覆盖率。
- **字体质量**：对于CJK语言，`use_localized_font_override` 应始终为 `true`，以确保正确的字符渲染。

## 验收标准

### 功能要求

**GIVEN** 游戏以任何语言运行，**WHEN** 玩家通过设置菜单切换到不同语言，**THEN** 所有UI元素、卡牌描述、系统消息和叙事文本立即更新到所选语言，而无需重新启动游戏。

**GIVEN** 语言包缺少翻译键，**WHEN** 系统尝试检索该键，**THEN** 它回退到主要语言（中文）并显示中文文本，同时为开发人员记录警告。

**GIVEN** 游戏处于显示多个UI元素的战斗中，**WHEN** 玩家切换语言，**THEN** 所有文本元素包括技能名称、描述、冷却计时器和状态效果都会原子性地更新，而不会导致战斗逻辑中断。

**GIVEN** 带有动态参数的文本（例如，"{general_name} 使用 {skill_name}"），**WHEN** 文本被本地化，**THEN** 参数以语法适当的方式正确替换为目标语言。

### 性能要求

**GIVEN** 游戏的冷启动，**WHEN** 加载语言包，**THEN** 在最低配置PC上加载时间不超过500ms。

**GIVEN** 所有三个语言包都已加载，**THEN** 额外的内存占用不超过5MB。

**GIVEN** 语言切换操作，**WHEN** 使用性能分析工具测量，**THEN** 在切换过程中帧时间增加不超过2ms。

### 资源管理要求

**GIVEN** 游戏的资源系统，**WHEN** 加载语言包，**THEN** 所有文本资源都通过标准资源加载系统加载，并在不再需要时正确卸载。

**GIVEN** 损坏或格式错误的语言包文件，**WHEN** 系统尝试加载它，**THEN** 系统优雅地回退到主要语言并记录适当的错误消息。

### UI/UX要求

**GIVEN** 带有文本内容的UI元素，**WHEN** 应用不同语言，**THEN** 文本正确适应容器边界，无剪裁、重叠或过多空白。

**GIVEN** 具有系统语言设置的玩家，**WHEN** 首次启动游戏，**THEN** 游戏根据系统设置自动选择适当的语言，并可手动覆盖。

**GIVEN** 特殊字符和Unicode符号，**WHEN** 以任何支持的语言显示，**THEN** 它们正确渲染，不显示方框、问号或其他替换字符。

### 测试要求

**GIVEN** 本地化系统实现，**WHEN** 运行单元测试，**THEN** 所有核心功能包括文本检索、语言切换、回退行为和参数替换都通过自动化测试验证。

**GIVEN** 集成测试场景，**WHEN** 本地化系统与UI、卡牌和地图系统一起测试，**THEN** 在活动游戏过程中更改语言时，所有文本元素都正确更新。

**GIVEN** 视觉验证过程，**WHEN** 对所有三种语言的关键UI屏幕进行截图，**THEN** QA审查员确认所有文本都根据语言特定的约定正确显示和格式化。

## 视觉/音频要求

### 视觉要求

1. **字体资源**：系统需要三种语言的独立字体资源
   - 中文：思源黑体（Source Han Sans）或类似CJK字体，支持简体中文字符集（约8000字符）
   - 英文：Roboto字体，优化拉丁字符显示
   - 日文：Noto Sans CJK JP，支持日文汉字、平假名、片假名

2. **字体回退机制**：当特定语言字体加载失败时，系统应使用Noto Sans作为默认字体，该字体支持所有CJK字符和拉丁字符

3. **字体大小调整**：由于不同语言的视觉密度不同，系统需要支持基于语言的字体大小缩放因子
   - 中文基准：100%
   - 英文基准：100%（可略缩小5-10% due to character density）
   - 日文基准：100%（接近中文密度）

4. **文本容器弹性**：所有包含本地化文本的UI容器必须预留足够的扩展空间，考虑到英文文本可能比中文长50-100%

5. **字符渲染质量**：确保所有字体在2D渲染中使用抗锯齿，避免字符边缘锯齿或模糊

### 音频要求

1. **语音文件命名规范**：语音文件必须遵循统一的命名约定以便系统查找
   ```
   voice_[content_type]_[language_key]_[unique_id].mp3
   示例：voice_card_swordmaster_zh.mp3
   ```

2. **语音文件组织**：语音文件应按照语言和内容类型组织在目录结构中
   ```
   assets/audio/voice/
   ├── zh/
   │   ├── cards/
   │   ├── ui/
   │   └── narration/
   ├── en/
   │   ├── cards/
   │   ├── ui/
   │   └── narration/
   └── ja/
       ├── cards/
       ├── ui/
       └── narration/
   ```

3. **语音与文本同步**：系统需要维护一个语音-文本映射表，确保语音文件与文本翻译对应正确的语言

4. **语音播放接口**：本地化系统应提供API让游戏逻辑能够根据当前语言选择合适的语音文件播放

## UI要求

### 核心UI集成要求

1. **语言选择界面**
   - 提供一个清晰的语言选择菜单，显示所有支持的语言选项
   - 显示当前选择的语言，使用该语言的本地化名称（如：中文、English、日本語）
   - 支持键盘导航和游戏手柄操作
   - 语言切换应立即生效，无需重启游戏

2. **文本绑定系统**
   - 所有UI控件必须通过语言键绑定文本，而非直接使用静态文本
   - 支持参数化文本替换，如"{general_name}使用了{skill_name}"
   - UI系统需要监听语言变更事件，并自动刷新所有绑定的文本

3. **动态布局调整**
   - UI容器必须支持根据文本长度自动调整大小
   - 对于可能超出容器边界的长文本，实现文本截断或自动换行
   - 提供最大文本长度限制，防止UI布局破坏

4. **字体和样式管理**
   - 每种语言应支持独立的字体设置，以优化字符渲染质量
   - 支持基于语言的字体大小调整，以适应不同语言的视觉密度
   - 确保所有UI元素在不同语言下保持一致的视觉样式和对齐方式

### 特定UI组件要求

1. **按钮和标签**
   - 支持多行文本显示和自动换行
   - 提供文本对齐选项（左对齐、居中、右对齐）
   - 支持文本颜色和样式变化以匹配游戏主题

2. **对话框和提示框**
   - 实现滚动文本显示以处理长文本内容
   - 支持富文本格式（粗体、斜体、颜色等）
   - 提供文本速度控制选项以适应不同玩家的阅读习惯

3. **设置菜单**
   - 语言设置选项应始终可见且易于访问
   - 提供语言变更预览功能，让用户在确认前看到效果
   - 保存用户的语言偏好设置并在下次启动时自动应用

4. **游戏内HUD元素**
   - 确保所有HUD文本元素（如生命值、技能名称、状态效果）正确本地化
   - 支持动态文本更新，如实时显示伤害数值或状态变化
   - 优化HUD布局以适应不同语言的文本长度变化
