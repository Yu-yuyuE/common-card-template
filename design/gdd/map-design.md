# 地图节点系统设计文档（Map Node System）

_Created: 2026-03-27_
_Status: Final v1_

---

## Overview

地图节点系统负责战役的宏观推进结构，提供树形路线选择、节点导航、资源消耗和事件触发功能。

**核心职责**：

- 生成并管理每位武将的专属战役地图（1场战役由3张树形小地图/3幕组成）
- 提供节点类型框架（战斗、精英、奇遇、商店、酒馆、军营）
- 追踪地图资源消耗（粮草）
- 协调各系统在节点层面的交互（触发战斗、商店、事件等）

**系统边界**：

- 本系统**不负责**战斗结算、卡牌管理、事件内容定义（由各子系统负责）
- 本系统**负责**地图数据结构、节点导航逻辑、资源消耗计算

---

## Player Fantasy

玩家体验“行军决策感”：

- 在有限补给下选择路线
- 在风险与收益节点中权衡
- 在历史事件中做策略抉择

## Detailed Rules

### 1) 一局（战役）地图结构

一局游戏（1场战役）包含 3 张小地图（3幕），对应起承转合：

- 幕1 起势图（教学+定流派）
- 幕2 扩张图（资源拉扯+事件密集）
- 幕3 决战图（机制拐点+Boss收束）

### 2) 每幕层数规则

- 层数范围：12~16
- 推荐默认：12层
- 分布建议：
  - 1-3层：基础战斗/轻事件
  - 4-9层：分歧路径（商店/酒馆/军营/奇遇）
  - 10-13层：高压战斗与精英
  - 14层：第一幕酒馆（在boss战前恢复体力），第二/三幕商店/酒馆/军营（由玩家决策升级卡/恢复体力/编排卡组）
  - 15层：Boss

### 3) 节点类型

1. 战斗节点
2. 精英战斗节点
3. 奇遇节点（历史随机事件）
4. 商店节点（买卡、买装备、升级卡，随机刷新）
5. 酒馆节点（恢复体力、恢复地图行动力）
6. 军营节点（更换兵种卡、升级兵种卡、移出卡组（编排卡组，包括所有类型））

### 4) 节点权重建议（每幕）

- 战斗：45%
- 奇遇：25%
- 商店：10%
- 酒馆：10%
- 军营：10%
- 精英：第一幕1~2个，第二幕3~4个，第三幕5~6个
- 强力：第一幕1~2个，第二幕2~3个，第三幕3~4个

### 5) 路线与分叉规则

- 每层提供2~3条分叉路径。
- 路径不能出现“无解断路”。
- 同一幕内保证至少出现1次商店、1次酒馆与1次军营。
- 同一层最多5个节点，最少1个节点。

### 6) 地图配置与数据结构

- **数据来源**：`hero_campaign_maps.csv`
- **字段定义**：
  - `hero_id`：武将唯一标识
  - `campaign_id`：战役编号（固定为1，每位武将1场战役）
  - `map_id`：小地图编号（1~3）
  - `terrain`：初始地形（平原/山地/森林/水域/沙漠/关隘/雪地）
  - `weather`：初始天气（晴/风/雨/雾）
  - `node_count`：节点总数（12~16）
  - `boss_node_id`：Boss节点编号
  - `starting_cargo`：初始粮草（150）
  - `historical_notes`：历史背景描述
- **生成规则**：每个武将拥有5场左右专属战役，每场战役包含3张小地图（3幕）

### 7) 节点导航与路径生成

- **树形结构**：每张小地图为树状结构，每个节点可有2~3个子节点
- **路径保证**：
  - 必须保证从起始节点到Boss节点存在至少一条路径
  - 使用深度优先搜索+回溯算法生成地图，确保无死路
  - 每张地图必须包含至少1个商店、1个酒馆、1个军营节点
- **节点生成算法**：
  1. 以起始节点为根
  2. 为当前节点随机选择2~3个子节点类型（按权重：战斗45%、奇遇25%、商店10%、酒馆10%、军营10%）
  3. 对于商店、酒馆、军营节点，确保每幕至少出现1次
  4. 每5层必须插入一个精英节点
  5. 第15层必须为Boss节点
  6. 检查所有节点是否可达，若存在不可达节点则重新生成

### 8) 资源消耗机制

- **资源**：粮草（Cargo）
- **初始值**：每张小地图开始时为150
- **消耗规则**：
  - 移动到任何非Boss节点消耗 2~8 粮草（随机值，平均5）
  - 移动到Boss节点消耗 10 粮草
  - 粮草值不能低于0，若移动后粮草<0，则拒绝移动并提示
- **补充机制**：
  - Boss战胜利后恢复 50 粮草
  - 酒馆节点可购买 40 粮草，可多次

### 9) 军营节点操作

- **功能**：允许玩家修改卡组（添加/移出卡组兵种卡、升级兵种卡）
- **约束**：
  - 兵种卡总数 ≤ 武将统帅值（3~6）
  - 添加新兵种卡时，系统检查当前总数是否已达上限，若已达上限则禁用添加按钮
  - 移出卡组操作后，该卡从卡组中永久移除（本场战役有效，进入下一场战役后恢复默认卡组）
  - 升级卡牌：消耗 30 粮草将Lv1兵种卡升级为Lv2（遵循 `barracks-system.md` 定义，需从 `troop_cards.csv` 中读取升级后效果）

### 10) 地形与天气动态变化

- **初始值**：由 `hero_campaign_maps.csv` 中的 `terrain` 和 `weather` 字段设定
- **触发机制**：
  - 奇遇节点：玩家选择后改变当前节点地形/天气
  - 卡牌效果：某些技能卡或诅咒卡可改变地形/天气
  - Boss机制：Boss战中可触发全局地形/天气变化
- **生效范围**：
  - 仅影响当前节点的战斗结算
  - 不影响已走过节点的战斗
  - 每次变化持续到下一节点

### 11) 地图状态持久化

- **存储内容**（保存在存档中）：
  - 当前战役进度（当前武将、当前小地图编号）
  - 当前粮草值
  - 已使用过的消耗卡列表（cardId, mapId）
  - 已升级的兵种卡列表（cardId）
  - 已访问的节点列表（避免重复访问）
- **恢复机制**：
  - 从存档加载时，重建地图状态
  - 重新应用已升级的兵种卡和已使用的消耗卡
  - 恢复当前粮草值

### 12) 节点交互流程

1. 玩家选择移动到下一节点
2. 系统计算消耗粮草，若不足则阻止移动
3. 若为战斗节点：
   - 加载战斗场景，传递当前地形/天气信息
   - 初始化敌人（从 `enemies.csv` 读取）
4. 若为商店节点：
   - 加载商店UI
   - 从 `troop_cards.csv`、`attack_cards.csv`、`skill_cards.csv` 随机生成可购买卡（考虑玩家已解锁卡）
   - 生成可购买的粮草（价格10~20）
5. 若为酒馆节点：
   - 恢复玩家部分粮草（购买40粮草）
6. 若为军营节点：
   - 加载卡组编辑界面
   - 显示当前卡组（包含兵种卡）
   - 允许玩家移出卡组卡牌或添加新兵种卡（受统帅值限制）
7. 若为奇遇节点：
   - 从 `event-system.md` 定义的事件库中随机选择一个事件
   - 显示事件描述和选项
   - 根据选择修改地图状态（如改变地形、获得资源、遭遇敌人）
8. 若为精英节点：
   - 加载精英战斗场景，敌人属性更强
9. 若为Boss节点：
   - 加载Boss战斗场景
   - 战斗胜利后恢复部分粮草
   - 标记当前小地图为完成
   - 进入下一小地图或战役

### 13) 系统交互边界

- **依赖**：
  - `resource-management-system.md`：提供粮草管理接口
  - `card-battle-system.md`：提供战斗触发接口
  - `enemies-design.md`：提供敌人数据
  - `event-system.md`：提供事件内容
  - `troop-cards-design.md`：提供兵种卡数据
  - `heroes-design.md`：提供武将统帅值
- **不依赖**：
  - 卡牌具体效果（由卡牌系统定义）
  - 事件具体内容（由事件系统定义）
  - 玩家卡组具体构成（由卡牌系统定义）

### 14) 错误处理与验证

- **编辑器验证**：
  - 检查每张小地图节点数是否在12~16范围内
  - 检查每张小地图是否包含至少1个商店、1个酒馆、1个军营
  - 检查每张小地图是否包含1个Boss节点
  - 检查所有节点是否可达
- **运行时验证**：
  - 防止玩家移动到不可达节点
  - 防止粮草为负时移动
  - 防止军营节点添加超过统帅值的兵种卡

## Formulas

### F1. 粮草消耗公式

`NextNodeCost = BaseCost + Random(0, Layer × 0.5)`

- **BaseCost**: 基础消耗 = 2
- **Layer**: 当前层数（1~15）
- **Random**: 随机值，范围 0 到 (Layer × 0.5) 取整
- **示例**: 第5层移动到下一节点，消耗 = 2 + random(0, 2.5) = 2~4

### F2. 酒馆节点恢复量

`RestoreAmount = BaseRestore`

- **BaseRestore**: 基础恢复 = 50

### F3. Boss战胜利奖励

`VictoryCargoReward = BaseReward`

- **BaseReward**: 基础奖励 = 50

### F4. 节点权重随机化

`NodeTypeChance = BaseWeight × MapModifier`

- **BaseWeight**: 各类节点的基准权重
  - 战斗节点: 0.45 (45%)
  - 奇遇节点: 0.25 (25%)
  - 商店节点: 0.10 (10%)
  - 酒馆节点: 0.10 (10%)
  - 军营节点: 0.10 (10%)
  - 精英节点: 0.05 (5%)
  - 强力节点: 0.05 (5%)
- **MapModifier**: 地图模式修正（默认为1.0）
- **约束条件**:
  - 同一幕内，商店、酒馆、军营至少出现1次
  - 精英节点数量：第一幕1~2、第二幕3~4、第三幕5~6
  - 强力节点数量：第一幕1~2、第二幕2~3、第三幕3~4

### F5. 路径可达性验证（伪代码）

```
function isBossReachable(map):
    graph = buildGraph(map.nodes)
    start = map.startingNode
    boss = map.bossNode

    visited = []
    queue = [start]

    while queue not empty:
        node = queue.pop()
        if node == boss:
            return true
        if node not in visited:
            visited.append(node)
            for child in node.children:
                if child not in visited:
                    queue.append(child)

    return false  // Boss不可达
```

### F6. 消耗卡使用追踪

`CanUseConsumable(cardId, mapId) = !UsedThisMap(cardId)`

- **UsedThisMap(cardId)**: 当前小地图中该卡是否已被使用
- 数据存储：`MapConsumableUsage[cardId][mapId] = true`
- 切图后，`UsedThisMap` 查询自动重置

## Edge Cases

### 1. 连续10层无商店/酒馆节点

- **问题**: 玩家资源和体力可能耗尽，无法继续推进
- **处理**: 生成地图时检测此情况，强制在第10层插入商店或酒馆节点

### 2. Boss节点不可达

- **问题**: 地图生成算法可能导致Boss节点被包围在不可达区域
- **处理**:

1. 使用 F5 路径可达性验证算法检测
2. 若Boss不可达，重新生成当前地图
3. 最多尝试3次，若仍失败，向编辑器报错

### 3. 玩家卡在死路（所有分支耗尽粮草）

- **问题**: 玩家选择了一条路线，但后续节点需要消耗的粮草超过当前持有量
- **处理**:
  1. 检查所有未访问节点的移动消耗
  2. 若所有未访问节点的消耗 > 当前粮草，允许玩家使用"紧急补给"选项（消耗部分HP换取粮草，如扣除20HP换40粮草）
  3. 若玩家拒绝，则显示提示"当前粮草不足以继续，请尝试其他路线或选择紧急补给"

### 4. 军营操作后卡组超统帅

- **问题**: 玩家在军营节点添加新兵种卡后，卡组中兵种卡总数超过武将统帅值（3~6）
- **处理**:
  1. 拒绝保存修改，弹出提示"当前兵种卡总数为 X，但武将统帅值为 Y，请移出卡组 X-Y 张兵种卡"
  2. 禁用"保存修改"按钮，直到当前兵种卡数 ≤ 统帅值

### 5. 同一节点被重复访问

- **问题**: 树形结构的某些地图可能允许玩家返回之前的节点
- **处理**:
  1. 默认情况下不允许返回（单向树形结构）
  2. 若需要支持节点重访，则节点效果不重置（如酒馆不会再次恢复粮草）
  3. 在节点UI上显示"已访问"标签

### 6. 战斗失败后果

- **问题**: 玩家在某节点战斗失败，需要定义后续行为
- **处理**:
  1. 扣除战斗开始时消耗的粮草（若已计算）
  2. 玩家生命值恢复到战斗前状态
  3. 不回退到上一节点，而是回到当前节点的起始状态
  4. 提示"战斗失败，请重新选择路线"（相当于消耗粮草但未获得收益）

### 7. 地图生成时节点类型冲突

- **问题**: 某一行需要同时放置商店和酒馆，但每层最多只能有5个节点
- **处理**:
  1. 按优先级排序：商店 > 酒馆 > 军营 > 其他
  2. 若当前层已达到5个节点，优先级低的节点被移动到下一层
  3. 记录日志以供调试

### 8. 事件触发后地图状态异常

- **问题**: 某些奇遇事件可能改变地形或天气，导致后续节点的地形数据矛盾
- **处理**:
  1. 事件仅影响当前事件的节点，不修改其他节点
  2. 地图配置 CSV 中的初始地形/天气保持不变
  3. 在存档中存储临时修改，不修改 CSV 源数据

### 9. 存档损坏恢复

- **问题**: 存档读取时发现地图状态数据不完整（如某个节点的粮草消耗记录缺失）
- **处理**:
  1. 对于缺失的粮草消耗，假设为标准消耗（5）
  2. 对于缺失的节点访问记录，标记为未访问
  3. 向玩家显示"检测到存档数据不完整，部分节点可能需要重新访问"的提示

### 10. CSV 配置文件错误

- **问题**: `hero_campaign_maps.csv` 中某条记录的 `node_count` 超出12~16范围
- **处理**:
  1. 编辑器加载 CSV 时，自动检测此错误
  2. 拒绝加载该条记录，并向控制台输出错误信息："英雄 X 的战役 Y 数据错误：节点数量 Z 超出有效范围（12~16）"
  3. 使用默认配置（12层，起始节点+Boss节点）替换错误配置

## Tuning Knobs

- 每幕层数：12~16（推荐默认15）
- 每层分叉数：2~3
- 节点类型权重：战斗45%、奇遇25%、商店10%、酒馆10%、军营10%
- 精英节点数量：第一幕1~2、第二幕3~4、第三幕5~6
- 强力节点数量：第一幕1~2、第二幕2~3、第三幕3~4
- 基础粮草消耗：2（每移动到下一层）
- 粮草消耗增量：随层数线性增长（最高增加8，即总消耗范围2~10）
- 酒馆恢复基数：30+（基于武将等级，最高50）
- Boss战奖励基数：固定50（遵循资源管理系统定义）

## Acceptance Criteria

1. 地图编辑器可配置每位武将的战役，包含3张树形小地图
2. 每张小地图的节点数在12~16之间，且仅包含1个Boss节点
3. 每张小地图至少包含1个商店节点、1个酒馆节点和1个军营节点
4. 节点生成算法保证从起始节点到Boss节点存在至少一条可达路径
5. 粮草消耗计算正确：移动到非Boss节点消耗2~8粮草，Boss节点消耗10
6. 酒馆节点正确恢复40粮草（购买），Boss战胜利后正确奖励50粮草
